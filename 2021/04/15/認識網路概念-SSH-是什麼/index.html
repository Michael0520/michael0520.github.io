認識網路概念 - SSH 是什麼？

Posted on 2021-04-15 Edited on 2021-04-19 In Backend Views: Disqus:

什麼是 SSH ?

SSH， 全名是 Secure SHell protocol 的簡寫 (安全的殼程式協定)，是一個遠端管理的網路協定，能讓使用者(伺服器管理員)透過網路遠端控制或修改(e.g 執行 commend)伺服器(電腦)。

SSH 這項服務是為了取代 Telnet 而建立，由於 Telnet 並沒有加密機制會有資安問題，所以 SSH 使用加密機制確保所有在網路上的傳輸都是安全的，他同時也提供了認證遠端使用者的功能。

這個 SSH 協定，在預設的狀態中，本身就提供兩個伺服器功能：

  1. 一個就是類似 telnet 的遠端連線使用 shell 的伺服器，亦即是俗稱的 ssh ；
  2. 另一個就是類似 FTP 服務的 sftp-server ！提供更安全的 FTP 服務。

連線加密技術簡介

什麼是『資料加密』呢？
簡單的說，就是將人們看的懂得原始電子資料，經過一些運算，讓這些資料變成沒有意義的亂碼 (至少對人類來說)，
然後再讓這個亂碼在網路上面傳輸，而當使用者想要查閱這個資料時，再透過解密運算，
將這些亂碼反推出原始的電子資料。由於這些資料已經被重新處理過，
所以，即使資料在網際網路上被 cracker 監聽而竊取，他們也不容易就推算得出來原始資料內容的。

[翻譯蒟蒻] 轉自鳥哥:

鳥哥常常說，加密機制有點像是兩個人之間的火星語對話啦！如果你跟你的朋友約定好使用你們制訂的某種特別語言， 這個語言只對你們兩個有意義。那麼當你們兩人講話時，在旁邊的人聽到的只是一堆沒有意義的聲音，因為他們聽不懂啊！ 即使路人將你的聲音錄下來，只要他不知道你們的特殊用語，那他就不可能瞭解你們對話的內容囉。

了解不同的加密技術

SSH 明顯的優勢在於它之前是用來確保 host 跟 client 之間資料傳輸的安全性，host代表你嘗試去存取的遠端伺服器， client 代表你使用的電腦，而 SSH 使用了三種不同的加密技術。

  • 對稱加密
  • 非對稱加密
  • 雜湊函數

對稱加密

對稱加密是一個加密的演算法，這個演算法會使用一個密鑰在client端與host端之間去加密與解密訊息，任何擁有這個密鑰的人都可以解鎖訊息密傳送。

對稱加密通常被稱作共享鑰匙或共享秘密加密，通常只會使用的一個鑰匙去加解密，有時候會有一對鑰匙，並用其中一隻鑰匙去加密另一隻鑰匙。

對稱加密的鑰匙可以被使用在整個 SSH 協定的溝通中， Client 端跟 Server 端會透過彼此同意的方法去產生密鑰，這個密鑰絕對不能讓任何第三方知道，而產生密鑰的程序被「Key exchange algorithm」所執行。

由於這個演算法並不會直接傳送密鑰給 client 和 host ，而是讓 client 和 host 共同分享一串資料，並分別更改它，並透過這串資料去計算出密鑰，就算有第三方獲得這串資料，沒有「Key exchange algorithm」也無法計算出密鑰。

例外值得注意的是，每一次 SSH 的交談中，都會有一個「secret token」，這個「secret token」會在 client 端做認證前產生。

而一旦這個密鑰產生，所有在 client 和 host 之間傳送的封包都會透過這個密鑰加密起來，就連使用者透過 console 輸入的密碼也是，所以 client 的身分會被保護著以免被監聽。

這裡有多種不同的對稱式加密演算法，但不侷限一定要使用哪個，像是 AES、CAST128、Blowfish 等等，在建立安全連線之前， client 端跟 host 端會從一個經過排序好的演算法清單中，決定出要使用哪一個演算法，在 client 端最首選的加密方法是在 host 端中被使用為雙向加密的加密方法。

非對稱加密

不像對稱式加密，非對稱是使用兩個不同的鑰匙去加解密，這兩把鑰匙分別是公鑰「public key」和私鑰「private key」，而這兩把鑰匙通常會一起使用。

  • 公鑰，顧名思義就是能公開散布並且分享給所有人，雖然他的功能會與私鑰緊密相連著，但私鑰是不能透過公鑰去計算出來的。這兩個密鑰的關係有點複雜，若用 A機器的公鑰去加密的訊息，只能用A機器的私鑰去解密，而公鑰也無法解密他自己加密的訊息，也無法解密其他不相干私鑰的訊息。
  • 私鑰就必須保持私密，也就是說為了連謝的安全性，不能有不相干的其他人知道私鑰，而 SSH 整個連線的優勢就是私鑰永遠不會被揭露出去，私鑰的功能是負責解開被他對應公鑰所加密後的訊息，因此任何有能力用公鑰加密訊息的人，都會擁有對應的私鑰。

與一般認知的不同，非對稱式加密並不使用在整個 SSH 的溝通中，而是被使用於對稱式加密的「Key exchange algorithm」中。

在安全連線建立之前， client 跟 host 雙方會產生一對暫時性的公私鑰，並用彼此的私鑰去產生對稱式加密中的共向密鑰。

一旦對稱式加密溝通建立後， host 端會使用 client 端的公鑰去產生一組數字傳送給client並驗證client端的身分，如果client端能成功解開這組數字，代表他擁有建立連結所對應的私鑰，整個SSH的溝通才正式啟動。

雜湊函數

單向的雜湊函數是另一種被使用於 SSH 連線中的加密方法。

單向雜湊函數在某些意義不同於上述兩個加密方法，單向雜湊函數所產生的數字永遠不會被解密，他會透過你輸入的資料產生出一組固定長度且唯一的數值，且這組數值沒有明顯的趨勢能被利用，也就是說他很難被反向推導。

從輸入的資料產生出加密後的雜湊值並不難，但要從雜湊值反推回輸入的值幾乎是不可能，也就是說如果 client 端如果擁有正確的資料，那他就可以用雜湊值去比對他所擁有的值是否是正確的。

SSH 使用雜湊函數去驗證訊息的正確性，而雜湊函數被HMACs(Hash-based Message Authentication Codes)這個演法所完成，他確保host端所接受的指令沒有被任何方式給竄改過。

在選擇對稱式加密演算法的同時，也必須選擇合適的訊息驗證演算法(雜湊函數)，而選擇訊息驗證演算法的方式跟選擇對稱式加密演算法類似。

每一個被傳送的訊息都會包含一個MAC(驗證值)，這個MAC使用對稱式鑰匙、訊息封包序列號碼、跟訊息內容所計算出來，最後會在加在使用對稱式加密後的訊息外，透過封包傳送出去。

SSH協定如何使用這些加密技術

SSH 運作的方式是使用 client-server 模式的同時能讓兩個遠端的系統達到認證與資料加密的功能。

SSH 預設運作在 TCP port 22(雖然如果需要是可以改變port號碼)，host(sevrer)是透過 port 22 去接受訊息(或者其他SSH指定的port號碼)，SSH 透過認證機制建立與 client 端安全的連結並開啟正確的 shell 環境，如果驗證是成功的。

client 端必須透過 TCP三次交握法 建立與 server 的連結，同時還要確保是安全的對程式連結並驗證 client 身分是否與 server 上之前的紀錄匹配(通常會以 RSA key 作為身分紀錄)，最後ssh會提供驗證所需要的使用者身分。

要建立起連結有兩個步驟，第一雙方系統都必須在加密標準達成共識去保護未來的溝通，第二使用者進行使用者驗證，一旦身分能夠跟server的資料匹配，使用者就能進入 server。

Session Encryption Negotiation

當 client 端試著使用 TCP 協定去連上 server 端， server 端會提供一個加密的協定跟他能支援的版本，如果 client 端有能夠對應的協定與版本，連結就會採用被接受的協定去建立， client 端也會用 server 端的非對稱式公鑰去驗證server的正確性。

一旦連結建立後， client 與 server 端會使用「 Diffie-Hellman Key Exchange Algorithm」去產生對稱式金鑰，這個演算法讓 client 和 sevrer 獲得一個加密金鑰，之後能使用它去加密整個溝通的過程。

這裡說明一下這個演算法基礎的運作方式：
client 跟 server 會同意一個主要且非常的數值(prime number)，且這個數值沒有什麼共通且普遍的因素，這個數值被稱做「seed value」。

接下來 client 跟 server 會決定一個共用的加密方式，並利用一個特定的演算法更改 seed value 去產生另一組值，這些透過執行大量的運算在seed上的方法被稱為「加密產生器」，AES就是其中之一。

之後會各自分別產生另一個主要數值，這個數值會作為秘密私鑰在之後的互動中。

這個新產生的私鑰，會有一個能共享的數值跟一個加密演算法(e.g AES)，這些會被使用於計算出公鑰，並分享給其他電腦。
client與server會使用各自的私要跟其他人的公鑰和最原本彼此同意出來的主要數值(prime number)去產生最後的對稱式金鑰，這個金鑰client與server會分別在運算，但最後會產生相同的加密金鑰。

現在 client 端與 server 都有了共同的金鑰，可以用來進行對稱式加密在整個SSH的溝通中，這個金鑰能夠加密也能夠解密。
現在安全的對稱式加密溝通已經被建立了，接下來就是用做使用者認證。

Authentication User

在使用者進入 server 之前的最後一步就是去驗證使用者的身分，大部分的SSH使用者是使用密碼，使用者在輸入 username 後會被要求輸入密碼，這些身分資料會透過對稱式加密通道安全地傳送，所以說沒有任何機會能夠被其它人獲取這些資料。

雖然密碼能夠被加密，但這個方式在安全的考量上仍不被建議，因為有很多機器人可以透過暴力破解或一些預設密碼成功進入 server ，所以推薦另一個替代方案，SSH Key Pairs。

它是一組非對稱式的金鑰，能夠被使用於驗證使用者身分，但不需要輸入任何密碼。

Conclusion

深入地去了解 SSH 底層如何運作的能讓我們用資安的角度去看待這個技術，大部分的人認為資訊安全的處理過程非常複雜且難以理解，但其實它比你想得還簡單。如果你想知道電腦花多久再計算這些雜湊值或驗證使用者，其實它只花不到一秒的時間，事實上，大部分的時間都會花在資料在網路中的傳輸上。

希望這篇關於 SSH 的文章能夠幫助你看見，不同技術的組合能夠建立起強大的系統且不同技術都在其中扮演重要的角色，同時你也知道為何 SSH 一出現 Telnet 就成為了過去。
